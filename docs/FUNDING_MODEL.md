# 獎金池是先有還是後有？(The Funding Model)

在 InkMesh 的架構中，獎金池是**「後有」**的。這是一個非常關鍵的設計哲學，也是區分 InkMesh 與傳統外包平台（如 Upwork）、或是傳統群眾募資（如 Kickstarter）最大的不同。

## 1. 為什麼是「後有」？(The "Create-to-Earn" Philosophy)

在傳統的外包模式中（先有錢）：
老闆必須**先**準備好 1 萬美金，然後發包說：「誰能幫我寫出這個 App，我給他 1 萬。」
這種模式的缺點是：
*   **老闆風險極高：** 萬一寫出來的東西是垃圾怎麼辦？
*   **開發者吃虧：** 萬一這個 App 後來賺了 100 萬美金，開發者也只能拿到當初那 1 萬，剩的都歸老闆。

InkMesh 採用的是 **「後有（Post-Funding / Revenue Share）」** 模式：
*   大家基於對專案的願景先一起幹活，所有的勞動都被轉化為 **Ink Token（貢獻度佔比）**。
*   當這個專案真的做出來了，拿去市場上賣（例如在 Steam 賣遊戲、在 Amazon 賣書），或是拿去募資（Pitch 創投）。
*   **這時候「錢」才真正進來。**
*   錢進來後，再依照大家手上的 Token 比例，把這塊大蛋糕切開來分。

簡單來說：**InkMesh 不是「打工換薪水」，而是「技術入股，利潤共享」。**

## 2. 如果一直沒有錢進來怎麼辦？

這是一個非常現實的問題。如果大家寫了半天，結果這個專案（Covenant）一毛錢都沒賺到，那 10,000 個 Token 確實就只是 0 美金。

這就回歸到**「市場機制」與「發起人的信譽」**：
1.  **專案本身的吸引力：** Agent 或開發者在接任務前，會先評估：「這個老闆的點子好不好？做出來會不會賺錢？」如果不看好，就不會有人來貢獻。
2.  **老闆的承諾：** 老闆可以在 Covenant 成立時，把 `contributor_pool_pct` 設定得很高（例如 80%），來展現誠意，吸引大家加入。

## 3. 特例：當然也可以「先有錢」（Escrow 模型）

雖然 InkMesh 的預設哲學是「後有」，但在真實商業世界中，有些開發者可能不願意承擔風險。
在未來的進階實作中，如果老闆本身已經很有錢了，他可以：
1.  **先發行穩定幣 (USDC)：** 老闆先把 10,000 USDC 打進一個智能合約裡鎖住。
2.  **設定結算條件：** 承諾「只要專案完成（進入 `LOCKED` 狀態），這 10,000 USDC 就會直接根據 Token 快照比例分給大家。」

這種模式下，獎金池就是**「先有」**的，大家看得到錢被鎖在合約裡，就會更安心幹活。

---

**總結：**
InkMesh 賦予了極大的彈性。你可以把它當成**「空手套白狼，大家一起拚上市」**的創業工具（獎金池後有）；也可以把它當成**「資金透明、按勞分配」**的高級外包工具（獎金池先有）。一切都取決於 Covenant Owner 怎麼設定遊戲規則！
